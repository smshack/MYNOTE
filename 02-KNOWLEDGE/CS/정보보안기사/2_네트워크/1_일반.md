### ✅ 1. 네트워크 기본 개념

**네트워크(Network)**  
→ 여러 장치(컴퓨터, 서버, IoT 등)를 연결해서 데이터를 주고받는 구조

**목적**

- 데이터 공유
    
- 자원 공유 (프린터, 서버 등)
    
- 통신 (인터넷, 이메일 등)
    

---

### ✅ 2. 프로토콜 (Protocol)

→ 네트워크에서 **통신 규칙**

**대표 프로토콜**

- HTTP → 웹 통신
    
- HTTPS → 보안 웹 통신
    
- TCP → 신뢰성 있는 전송
    
- UDP → 빠른 전송 (신뢰성 낮음)
    
- IP → 주소 지정
    
- FTP → 파일 전송
    
- DNS → 도메인 → IP 변환
    

👉 쉽게 말하면  
“서로 약속된 방식으로 대화하는 규칙”

---

### ✅ 3. 거리에 따른 네트워크 분류

| 구분  | 설명                            |
| --- | ----------------------------- |
| LAN | Local Area Network (집, 회사 내부) |
| MAN | Metropolitan (도시 단위)          |
| WAN | Wide Area (국가, 인터넷)           |

---

### ✅ 4. 전송 방식에 따른 분류

### 🔹 1) 회선 교환 방식 (Circuit Switching)

- 통신 전에 **전용 회선 확보**
    
- 예: 전화
    

✔ 특징

- 안정적
    
- 지연 적음
    
- 비효율 (회선 독점)
    

---

### 🔹 2) 패킷 교환 방식 (Packet Switching)

- 데이터를 **작은 패킷으로 나눠 전송**
    
- 인터넷 방식
    

✔ 특징

- 효율적
    
- 경로 동적 변경
    
- 지연 가능성 있음
    

👉 현재 인터넷 = 대부분 패킷 방식

| 1 물리    | 네트워크 인터페이스 | 전기신호         | 케이블, 광케이블                   |

### ✅ 7. 면접 한방 정리 (🔥 진짜 중요)

👉  
클라이언트는 HTTP 요청을 생성하고 TCP로 연결을 맺은 뒤  
IP를 통해 목적지를 찾고 ARP로 MAC 주소를 알아내어 전송합니다.

데이터는 계층을 내려가며 캡슐화되고,  
수신 측에서는 반대로 디캡슐화되어 최종적으로 응용 계층에서 처리됩니다.

좋아, 이건 **웹/백엔드/인프라 면접 핵심 세트**라서  
흐름 + 구조 + 표로 정확하게 정리해줄게.

### ✅ 1. HTTP 세션(연결) 과정

## 🔷 전체 흐름 (브라우저 → 서버)

```
1. URL 입력
   ↓
2. DNS 조회 (도메인 → IP)
   ↓
3. TCP 연결 (3-way handshake)
   ↓
4. HTTP 요청(Request) 전송
   ↓
5. 서버 처리
   ↓
6. HTTP 응답(Response) 반환
   ↓
7. 연결 유지 or 종료
```

---

## 🔷 핵심 포인트

- HTTP는 **Stateless (상태 없음)**
    
- 상태 유지를 위해 **쿠키 / 세션 / 토큰 사용**
    
- HTTP/1.1 → 기본 **Keep-Alive (연결 유지)**


|---|---|
|Host|요청 서버|

## 🔷 특징

- 헤더 + 바디 구조
    
- GET은 보통 Body 없음
    
- POST/PUT은 Body 포함

### ✅ 4. HTTP 요청 방식 (Method)

## 🔷 주요 메서드

|---|---|---|

    - POST: Body 사용, 보안성 상대적으로 높음

### ✅ 7. 핵심 한방 정리 (면접용🔥)

👉  
HTTP는 클라이언트와 서버 간 데이터를 주고받는 프로토콜로,  
요청(Request)과 응답(Response) 구조로 동작합니다.

요청은 Method, Header, Body로 구성되고  
응답은 상태 코드와 Header, Body로 구성됩니다.

HTTP는 stateless이기 때문에  
쿠키, 세션, 토큰을 통해 상태를 유지합니다.
---
### ✅ 1. 클라이언트 측 정보 유지 방법

HTTP는 **Stateless(무상태)**라서  
요청 간 사용자 상태를 유지하려면 별도 방법이 필요함.

## 🔷 1) 쿠키 (Cookie)

**개념**

- 브라우저에 저장되는 데이터
    
- 서버가 내려주고 이후 요청마다 자동 포함
    

**특징**

- 클라이언트에 저장됨
    
- 자동 전송됨
    

**사용 예**

- 로그인 유지
    
- 장바구니
    

---

## 🔷 2) 세션 (Session)

**개념**

- 서버에 사용자 상태 저장
    
- 클라이언트는 **세션 ID만 보유 (쿠키로 전달)**
    

**특징**

- 서버 저장 → 보안 높음
    
- 메모리 사용 증가
    

---

## 🔷 3) 토큰 (JWT 등)

**개념**

- 서버가 사용자 정보를 토큰으로 만들어 전달
    
- 클라이언트가 저장 후 요청 시 전송
    

**특징**

- 서버 상태 저장 필요 없음 (Stateless)
    
- 확장성 좋음
    

---

## 🔷 비교 정리

| 방식  | 저장 위치 | 특징     | 보안  |
| --- | ----- | ------ | --- |
| 쿠키  | 클라이언트 | 자동 전송  | 낮음  |
| 세션  | 서버    | 안정적    | 높음  |
| 토큰  | 클라이언트 | 확장성 좋음 | 중간  |

---

### ✅ 2. 쿠키 보안 옵션 (🔥 중요)

쿠키는 **클라이언트에 저장되기 때문에 보안 설정 필수**

---

## 🔷 주요 옵션

### 1) HttpOnly

**설명**

- JavaScript에서 쿠키 접근 불가
    

**목적**

- XSS 공격 방지
    

👉 document.cookie 접근 차단

---

### 2) Secure

**설명**

- HTTPS에서만 전송
    

**목적**

- 중간자 공격 방지
    

---

### 3) SameSite

**설명**

- 다른 사이트 요청 시 쿠키 전송 여부 제어
    

**옵션**

| 값      | 설명                |
| ------ | ----------------- |
| Strict | 완전 차단             |
| Lax    | 일부 허용 (기본)        |
| None   | 모두 허용 (Secure 필요) |

**목적**

- CSRF 방지
    

---

### 4) Domain

**설명**

- 쿠키가 적용되는 도메인 지정
    

---

### 5) Path

**설명**

- 쿠키가 적용되는 URL 경로
    

---

### 6) Max-Age / Expires

**설명**

- 쿠키 만료 시간 설정
    

---

### ✅ 3. 보안 옵션 정리 표

| 옵션          | 목적       | 방어 공격 |
| ----------- | -------- | ----- |
| HttpOnly    | JS 접근 차단 | XSS   |
| Secure      | HTTPS 전송 | MITM  |
| SameSite    | 요청 제한    | CSRF  |
| Max-Age     | 만료       | 세션 탈취 |
| Domain/Path | 범위 제한    | 쿠키 오용 |

---

### ✅ 4. 안전한 쿠키 설정 예시

```id="cookie-secure"
Set-Cookie: sessionId=abc123;
 HttpOnly;
 Secure;
 SameSite=Strict;
 Path=/;
 Max-Age=3600;
```

👉 실무에서는 거의 이 조합 사용

---

### ✅ 5. 면접 핵심 포인트 (🔥)

👉  
HTTP는 Stateless이기 때문에  
쿠키, 세션, 토큰을 사용해 상태를 유지합니다.

특히 쿠키는 클라이언트에 저장되므로  
HttpOnly, Secure, SameSite 옵션을 통해  
XSS, CSRF 같은 공격을 방어해야 합니다.
